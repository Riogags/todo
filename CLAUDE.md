# CLAUDE.md — Smart Irrigation System (ML Component)

## What this project is

A smart irrigation system built for a university engineering project (Electrical/Electronic Engineering). It has a **hardware prototype** (sensors + microcontroller + actuator/pump) and a **machine learning component**. This repo is the ML component.

The ML model takes live sensor readings and decides whether the field should be irrigated.

## The ML task — read this carefully

- **Task type:** Binary classification.
- **Algorithm:** Logistic Regression. (Do not use linear regression — the target is a category, not a number.)
- **Prediction target:** `Irrigate` → `Yes` / `No` (encode as `1` / `0`).

## Features (model inputs)

Exactly three, all measurable by the hardware team’s sensors:

|Feature        |Unit|Sensor (hardware side)                   |Notes                                             |
|---------------|----|-----------------------------------------|--------------------------------------------------|
|`soil_moisture`|%   |capacitive/resistive soil moisture sensor|Most important feature                            |
|`temperature`  |°C  |DHT11/DHT22                              |Affects evaporation/transpiration                 |
|`air_humidity` |%   |DHT11/DHT22                              |Low humidity → faster water loss → irrigate sooner|


> Feature importance for irrigation timing roughly: soil_moisture ≫ temperature ≈ air_humidity.
> Temperature and air_humidity both come from a single DHT11/DHT22 sensor, so this is cheap on the hardware side.
> Soil pH is NOT used in this version.

## Target

`irrigate` — binary. `1` = irrigate, `0` = don’t.

Example dataset shape:

|soil_moisture|temperature|air_humidity|irrigate|
|-------------|-----------|------------|--------|
|18           |34         |30          |1       |
|25           |31         |38          |1       |
|45           |28         |70          |0       |
|52           |26         |75          |0       |

## What to build

1. **Data loading & preprocessing**
- Load CSV (`data/irrigation.csv`).
- Handle missing values, scale/standardize features (StandardScaler).
- Train/test split (e.g. 80/20, `random_state` fixed for reproducibility).
1. **Model training**
- `sklearn.linear_model.LogisticRegression`.
- Fit on training set.
1. **Evaluation**
- Accuracy, precision, recall, F1.
- Confusion matrix (print + save a plot).
- Show the decision boundary / coefficients so the model is explainable.
1. **Persistence**
- Save trained model + scaler (`joblib`) so the hardware team can load and run inference.
- Provide a `predict(soil_moisture, temperature, air_humidity) -> "Yes"/"No"` helper for deployment.
1. **Inference script**
- A small script the microcontroller-side/host can call with three sensor values and get a decision.

## Tech stack

- Python 3.x
- `pandas`, `numpy`, `scikit-learn`, `joblib`, `matplotlib`
- Keep it lightweight — this may run on a Raspberry Pi or be exported for an embedded host.

## Build decisions (as implemented)

These are the defaults chosen for the open items above. They are easy to revisit.

- **Dataset:** No real labelled dataset was supplied, so `data/irrigation.csv` is **synthetic but physically reasonable**, generated by `src/generate_data.py` (fixed seed). Replace it with real field data and re-run `src/train.py` — nothing else changes.
- **Inference interface:** Exposed as a Python **function call** — `predict(soil_moisture, temperature, air_humidity) -> "Yes"/"No"` in `src/predict.py` — plus a CLI for quick host/serial integration. The hardware team can `from predict import predict` and call it directly.
- **Threshold logic:** **Model + optional hard safety rule.** By default the decision is model-only. A configurable safety override (`never irrigate when soil_moisture >= SAFETY_MOISTURE_CEILING`) can be enabled so the pump can never fire on already-wet soil regardless of the model.

## Optional second model — water/sunlight estimator (Linear Regression)

A separate, optional model that estimates **continuous amounts** from the same
three sensor inputs:

- `water_liters_per_m2` — recommended water to apply (litres per m² per day).
- `sunlight_hours` — **advisory** recommended daily sun-exposure (a field can't
  actuate the sun; reframe as an actuated output if this is a greenhouse with
  grow-lights).

This uses `sklearn.linear_model.LinearRegression` (multi-output). That does
**not** contradict the "no linear regression" rule above: that rule is about the
`irrigate` target, which is a *category* (classification → Logistic Regression).
Water/sunlight are *numbers*, so regression is the correct tool. Right tool per
target.

Files: `src/generate_water_sunlight_data.py`, `src/train_regression.py`,
`src/estimate.py`. Dataset `data/water_sunlight.csv` is **synthetic** (fixed
seed) — replace with real agronomy/field measurements and re-run.

> In the synthetic run, water fits well (R² ≈ 0.88) but sunlight fits poorly
> (R² ≈ 0.48) — expected, because sunlight need is mostly a crop property, not a
> function of these three sensors.

## How to run

```bash
pip install -r requirements.txt

# Model 1 — irrigate decision (classification)
python src/generate_data.py     # (re)create the synthetic dataset
python src/train.py             # train, evaluate, save model + scaler + plot
python src/predict.py 18 34 30  # -> Yes
python src/predict.py 52 26 75  # -> No

# Model 2 — water/sunlight estimate (regression, optional)
python src/generate_water_sunlight_data.py  # (re)create its synthetic dataset
python src/train_regression.py              # train, evaluate, save regressor
python src/estimate.py 18 34 30             # -> water_liters_per_m2 + sunlight_hours
```
